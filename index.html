About Me

일하는 방식과 디자인 관점

일하는 방식과 디자인을 바라보는 관점을 질문과 답변 형식으로 정리했습니다.

Q1
어떤 문제를 해결하는 디자이너인가요?
복잡한 시스템과 데이터를 사용자가 이해하고 활용할 수 있는 구조로 정리하는 데 강점이 있습니다. SaaS Admin, BI Dashboard, Data Platform처럼 정보량이 많고 역할이 복잡한 제품에서 화면 구조, 정보 위계, 컴포넌트 사용 기준을 명확히 설계합니다.
Q2
디자인 시스템을 어떻게 바라보나요?
디자인 시스템은 단순히 UI 컴포넌트를 모아둔 라이브러리가 아니라, 제품을 일관되게 만들기 위한 운영 기준이라고 생각합니다. 컴포넌트의 역할, 사용 조건, 상태, 문서화 기준을 정리해 디자이너와 개발자가 같은 기준으로 협업할 수 있도록 만드는 데 집중합니다.
Q3
일할 때 중요하게 생각하는 방식은 무엇인가요?
문제를 바로 시각화하기보다 먼저 구조를 이해하고 기준을 세우는 방식을 중요하게 생각합니다. 최근에는 AI를 활용해 가이드라인 작성, UX Writing, 반복적인 문서화 작업을 효율화하며 팀이 더 빠르게 검증하고 개선할 수 있는 workflow를 만들고 있습니다.

& Documentation

디자인 시스템의 컴포넌트 구조와 사용 기준을 재정리하고, 개발 문서와 연결 가능한 운영 기반을 구축한 프로젝트입니다.

Background

기존 디자인 시스템은 컴포넌트 구조와 사용 기준이 일부 분산되어 있어, 디자이너와 개발자가 동일한 기준으로 컴포넌트를 사용하기 어려운 문제가 있었습니다.

01
분산된 컴포넌트 기준
컴포넌트 역할과 사용 조건이 명확히 정리되어 있지 않아 화면마다 다른 방식으로 사용될 가능성이 있었습니다.
02
문서화 기준 부족
Overview, Anatomy, Usage, Do / Don't 등 가이드라인 작성 방식이 통일되어 있지 않았습니다.
03
개발 연계성 부족
디자인 기준이 개발 문서와 자연스럽게 연결되기 어려워 구현 및 유지보수 관점의 정리가 필요했습니다.

Goal

컴포넌트 구조를 재정리하고 사용 기준을 문서화하여, 디자인 시스템의 일관성, 재사용성, 개발 연계성을 높이는 것을 목표로 했습니다.

01
Consistency
디자이너가 동일한 기준으로 컴포넌트를 선택하고 사용할 수 있도록 구조를 정리합니다.
02
Reusability
중복되거나 역할이 모호한 컴포넌트를 재분류하여 재사용 가능한 시스템 구조를 만듭니다.
03
Dev Alignment
문서 사이트와 개발 환경을 연결할 수 있는 기반을 검토하고 협업 프로세스를 정리합니다.

My Role

디자인 시스템 구조 개선부터 컴포넌트 가이드라인 작성, 문서 사이트 구축 환경 테스트, 개발 연동 검토까지 담당했습니다.

01
System Structure
기존 컴포넌트 구조를 분석하고, 그룹과 네이밍 체계를 재정리했습니다.
02
Guideline Writing
주요 컴포넌트의 Overview, Anatomy, Usage, Do / Don't 기준을 작성했습니다.
03
Foundation Standard
Text Style, Variable, Spacing 등 디자인 시스템의 기반 기준을 정리했습니다.
04
Documentation Test
Docusaurus 기반 문서 사이트 구축을 테스트하고 GitHub / Vercel 연동 가능성을 검토했습니다.

Process

기존 구조를 분석한 뒤 컴포넌트 체계, Foundation 기준, 문서화 방식, 개발 연동 가능성을 순차적으로 정리했습니다.

01
Analyze Component Structure
기존 컴포넌트 구조와 사용 기준을 점검하고, 중복되거나 역할이 불명확한 요소를 파악했습니다.
02
Reorganize Component Groups
컴포넌트의 역할과 사용 목적을 기준으로 그룹을 재분류하고 네이밍 체계를 정리했습니다.
03
Define Foundation Standards
Text Style, Variable, Spacing 등 디자인 시스템의 기반 요소를 정리했습니다.
04
Write Component Guidelines
주요 컴포넌트별 사용 조건, 구조, 상태, Do / Don't 기준을 문서화했습니다.
05
Test Documentation Site
Docusaurus 기반 문서 사이트 구축을 테스트하고, 문서 구조와 페이지 구성을 검토했습니다.
06
Review Dev Workflow
GitHub / Vercel 연동을 통해 디자인 문서와 개발 문서가 연결될 수 있는 협업 방식을 정리했습니다.

Outcome

디자인 시스템의 컴포넌트 사용 기준을 명확히 하고, 향후 개발 문서와 연동 가능한 운영 기반을 마련했습니다.

01
Clear Usage Standard
컴포넌트별 역할과 사용 기준을 정리하여 디자이너가 일관된 기준으로 UI를 설계할 수 있는 기반을 만들었습니다.
02
Scalable Documentation
Overview, Anatomy, Usage, Do / Don't 형식의 문서화 기준을 정리하여 이후 컴포넌트 확장에 대응할 수 있도록 했습니다.
03
Better Dev Collaboration
Docusaurus, GitHub, Vercel 기반의 문서 사이트 운영 가능성을 검토하여 개발 협업과 연결 가능한 구조를 마련했습니다.

Before & After

Before

  • ✕컴포넌트 구조와 사용 기준이 일부 분산됨
  • ✕문서화 기준이 일관되지 않음
  • ✕디자인 기준과 개발 문서의 연결성이 부족함

After

  • ✓컴포넌트 그룹과 네이밍 기준을 재정리
  • ✓Usage Guideline 작성 기준을 통일
  • ✓문서 사이트 기반의 운영 및 개발 연동 가능성 검토

Key Takeaway

이 프로젝트를 통해 디자인 시스템은 단순한 UI 라이브러리가 아니라, 제품의 일관성과 협업 효율을 높이는 운영 체계라는 점을 더 깊이 이해했습니다. 컴포넌트 구조, 사용 기준, 문서화 방식, 개발 연동 흐름을 함께 고려하며 시스템을 설계하는 경험을 쌓았습니다.

Background

SSP는 내부 Admin 플랫폼을 넘어 외부 솔루션과 연동 가능한 서비스 플랫폼으로 확장 가능성을 검토하고 있었습니다. 이를 위해 Space Vision, Trace it, Seerslab 등 다양한 외부 솔루션과의 연동 논의가 진행되었고, 사업 검토와 개발 협업을 위한 POC 화면이 필요했습니다.

외부 솔루션의 기능을 SSP 안에서 어떻게 연결하고, Admin 사용자가 어떤 정보를 확인하고 관리할 수 있을지 화면 기준으로 구체화하는 과정이 필요했습니다.

Problem

외부 솔루션 연동 논의는 진행되고 있었지만, 실제 서비스 플랫폼 안에서 어떤 화면과 기능으로 연결될지에 대한 기준은 아직 구체화되지 않은 상태였습니다.

01
연동 화면 기준 부족
외부 솔루션을 SSP에 어떻게 연결하고 Admin에서 어떻게 보여줄지 화면 기준이 필요했습니다.
02
데이터 확인 범위 정의 필요
솔루션 연동 후 Dashboard에서 어떤 데이터를 확인하고 관리해야 하는지 정리가 필요했습니다.
03
공간 기반 관리 경험 필요
Map View를 통해 매장, 오브젝트, 공간 정보를 직관적으로 확인할 수 있는 구조가 필요했습니다.
04
데모 시나리오 부족
전시 환경에서 기능 가치를 쉽게 전달할 수 있는 사용 흐름과 시연 시나리오가 필요했습니다.
05
협업 방향 정렬 필요
외부 협력사, 내부 개발팀, 영상 제작사 간 구현 범위와 화면 방향을 맞추는 과정이 필요했습니다.

Goal

외부 솔루션과 SSP의 연동 구조를 화면으로 구체화하고, Dashboard / Map View POC를 통해 데이터 확인 및 공간 기반 관리 경험을 시각화하는 것을 목표로 했습니다.

01
Visualize Integration Flow
외부 솔루션이 SSP 안에서 어떤 기능과 화면으로 연결될 수 있는지 구체화합니다.
02
Design POC Experience
Dashboard와 Map View를 통해 데이터 확인, 공간 정보 탐색, 오브젝트 관리 흐름을 시각화합니다.
03
Support Business Discussion
신규 사업 검토와 외부 협업 논의에 활용할 수 있는 화면 산출물과 데모 시나리오를 마련합니다.
04
Align with Development
Unity 개발, View/Edit 기능 검토, Admin 기능 설계를 통해 실제 구현 방향과 디자인 방향을 정렬합니다.

My Role

Product Designer이자 POC UX Designer로서 외부 솔루션 연동 논의부터 Dashboard / Map View POC 화면 설계, Unity 개발 대응, 전시 데모 시나리오 기획, Tenant Management Admin 설계까지 프로젝트 전반의 UX/UI 구체화를 담당했습니다.

01
POC UX Planning
외부 솔루션 연동 가능성을 검토하고, SSP 내에서 필요한 화면과 기능 흐름을 정리했습니다.
02
Dashboard POC
연동 데이터를 Admin에서 빠르게 확인할 수 있도록 Dashboard POC 화면 구조 검토에 참여했습니다.
03
Map View POC Design
공간 기반 정보를 확인하고 관리할 수 있는 Map View 화면과 View/Edit 기능 구조를 디자인했습니다.
04
Development Collaboration
Unity 기반 개발 대응, 기능 범위 확인, 개발 결과물 검토를 통해 디자인과 구현 방향을 맞췄습니다.
05
Demo Scenario Planning
Picking, RFID, Load-cell 등 전시 데모 컨셉을 기반으로 사용 흐름과 시연 시나리오를 정리했습니다.
06
Tenant Management UX/UI
SSP Admin 내 Tenant Management 기능의 List / Detail 화면 구조를 기획하고 디자인했습니다.

Process

외부 솔루션 도입 논의에서 시작해 SSP 연동 가능 기능을 검토하고, Dashboard / Map View POC 화면과 전시 데모 시나리오, Admin 관리 기능까지 단계적으로 구체화했습니다.

01
External Solution Discussion
Space Vision, Trace it, Seerslab 등 외부 솔루션의 SSP 도입 논의에 참여하고 연동 방향을 검토했습니다.
02
Integration Scope Review
외부 솔루션이 SSP 내에서 어떤 기능으로 연결될 수 있는지 기능 범위와 데이터 흐름을 확인했습니다.

Dashboard POC Structure
캠페인, 매장, POS Data 등 연동 가능한 데이터를 Admin 화면에서 어떻게 확인할지 Dashboard POC 구조를 검토했습니다.
04
Map View POC Design
공간, 매장, 오브젝트 정보를 기반으로 Map View 화면 구조와 정보 위계를 설계했습니다.
05
View / Edit Function Review
Map 기반 View / Editor 기능 구현 방향을 검토하고 Unity 개발 대응을 진행했습니다.
06
Demo Scenario Planning
Picking, RFID, Load-cell 컨셉을 기반으로 전시 데모 시나리오와 화면 흐름을 정리했습니다.

Tenant Management Design
SSP Admin에서 Tenant 정보를 관리할 수 있도록 List / Detail 중심의 관리 화면을 기획하고 디자인했습니다.

Feedback & Alignment
외부 협력사, 개발팀, 영상 제작사, STI 개발 담당자와 피드백을 주고받으며 구현 방향을 조율했습니다.

Key Work

POC 화면 설계뿐 아니라 외부 솔루션 연동 검토, Unity 개발 대응, 전시 데모 시나리오, Admin 관리 기능 설계까지 SSP 확장 가능성을 검토하기 위한 여러 산출물을 함께 정리했습니다.

01
Space Vision / Trace it 연동 검토

Space Vision, Trace it 등 외부 솔루션의 SSP 도입 논의에 참여하여 서비스 연동 방향과 기능 흐름을 검토했습니다. 외부 솔루션이 SSP 내에서 어떻게 관리되고 확장될 수 있는지 확인하며, POC 화면 제작과 Admin 기능 설계를 위한 기반을 정리했습니다.

  • —Space Vision SSP 도입 논의 플로우업
  • —Trace it ↔ SSP 도입 논의 플로우업
  • —Trace it ↔ SSP 도입 관련 개발 미팅 참여
  • —Space Vision 캠페인 ↔ POS Data 연동 후속 논의 검토
  • —외부 솔루션과 SSP 간 기능 연결 흐름 검토
02
Seerslab 협업 및 Unity 개발 대응

Unity 기반 Map View 개발 대응과 기능 검토를 진행했습니다. 외부 협력사와 내부 개발팀 사이에서 화면 구현 방향과 기능 범위를 확인하고, 디자인 산출물이 실제 개발 결과물로 연결될 수 있도록 커뮤니케이션을 지원했습니다.

  • —Unity 개발 대응
  • —개발 일정 및 기능 범위 확인
  • —View / Editor 기능 구현 방향 검토
  • —개발 결과물 확인 및 커뮤니케이션 대응
03
Dashboard POC 제작 참여

SSP 연동 서비스의 핵심 데이터를 빠르게 확인할 수 있도록 Dashboard POC 제작에 참여했습니다. 캠페인, 매장, POS Data 등 연동 가능한 데이터를 Admin 화면에서 어떻게 시각화하고 탐색할지 검토하며 데이터 기반 관리 화면의 초기 방향성을 구체화했습니다.

  • —Dashboard POC 제작 참여
  • —데이터 확인 목적의 화면 구조 검토
  • —캠페인, 매장, POS Data 등 연동 데이터 표현 방식 검토
  • —Admin 사용자가 주요 데이터를 빠르게 파악할 수 있는 구조 설계 지원
04
Map View POC 디자인

공간 기반 정보를 직관적으로 확인할 수 있는 Map View POC를 디자인했습니다. View/Edit 기능 구조를 기준으로 맵 영역, 정보 패널, 오브젝트 관리 흐름을 설계하고, 향후 Editor 기능으로 확장 가능한 UI 구조를 검토했습니다.

  • —Map View POC 디자인
  • —공간 / 매장 / 오브젝트 기반 정보 표시 방식 검토
  • —View / Edit 기능 기준 화면 설계
  • —Map 기반 관리 화면의 정보 위계 정리
  • —향후 Editor 기능 개발 대응을 위한 UI 구조 검토
05
전시 데모 시나리오 기획 참여

전시 환경에서 SSP 연동 기능을 효과적으로 보여줄 수 있도록 데모 시나리오 기획에 참여했습니다. Picking, RFID, Load-cell 등 주요 컨셉을 기반으로 사용 흐름을 정리하고, 외주 개발사 및 영상 제작 대응을 통해 데모 완성도를 높이는 데 기여했습니다.

  • —STI 개발 대응

Background

SSP Admin은 테넌트, 회사, 데이터 구조를 관리해야 하는 SaaS 운영 플랫폼입니다. 기능별 요구사항은 존재했지만, 실제 사용자가 정보를 탐색하고 설정을 변경하는 화면 구조와 관리 흐름은 구체화가 필요한 상태였습니다.

Admin 사용자가 복잡한 조직·테넌트·데이터 정보를 쉽게 탐색하고 관리할 수 있도록, 기능별 IA와 화면 구조를 정리하는 과정이 필요했습니다.

Problem

SaaS Admin 관리 기능은 여러 도메인으로 나뉘어 있었고, 각 기능의 정보 구조와 사용 흐름을 일관된 기준으로 정리할 필요가 있었습니다.

01
관리 기능 IA 기준 부족
Tenant, Company, Data 관리 기능의 정보 구조가 명확히 정리되지 않아 메뉴와 화면 흐름을 재구성할 필요가 있었습니다.
02
Detail 화면 탭 구조 필요
Tenant Detail 화면에서 General, Store, Design 등 정보를 어떤 기준으로 구분하고 탐색하게 할지 기준이 필요했습니다.
03
복잡한 데이터 관리 구조
Data Management는 데이터 구조가 복잡해 Admin 사용자가 이해하기 쉬운 메뉴 설계와 유저 플로우가 필요했습니다.
04
변경사항 반영 및 화면 개선 필요
Company Management 화면의 기획 누락과 변경사항을 점검하고, 일관된 UI 패턴으로 반영할 필요가 있었습니다.
05
개발 구현 검수 필요
개발된 View/Edit 기능이 디자인 산출물과 일치하는지 확인하고, 구현 품질을 검수하는 과정이 필요했습니다.

Goal

Admin 사용자가 테넌트, 회사, 데이터 정보를 쉽게 탐색하고 관리할 수 있도록 기능 구조를 정리하고, 개발 가능한 수준의 화면 설계와 유저 플로우를 제공하는 것을 목표로 했습니다.

01
Structure Admin IA
Tenant, Company, Data 관리 기능을 사용 목적과 정보 구조에 맞게 분류하고 메뉴 체계를 정리합니다.
02
Design Management Flow
List, Detail, Tab 구조를 기반으로 Admin 사용자가 필요한 정보를 탐색하고 설정을 변경할 수 있는 흐름을 설계합니다.
03
Clarify Data Management
복잡한 데이터 관리 개념을 Data Ingestion 유저 플로우와 화면 단위로 구체화합니다.
04
Align with Development
개발팀 요구사항을 확인하고 구현 화면을 검수하여 디자인 산출물이 실제 제품 구현까지 연결되도록 합니다.

My Role

Product Designer이자 Admin UX Designer로서 SaaS Admin 관리 기능의 IA 설계, 화면 기획, UX/UI 디자인, 데이터 관리 유저 플로우 작성, 개발 리뷰 및 구현 화면 검수까지 담당했습니다.

01
Admin IA Design
Tenant, Company, Data Management 기능을 분류하고, Admin 사용자가 탐색하기 쉬운 메뉴와 화면 구조를 설계했습니다.
02

03
Data Management UX
SaaS 데이터 관리 솔루션을 리서치하고, Data Ingestion 중심의 유저 플로우와 화면 구조를 설계했습니다.
04

Company Management 화면의 기획 누락과 변경사항을 점검하고, 컴포넌트 및 UI 구조를 개선했습니다.
05
Development Review
개발팀 요구사항을 확인하고 기획 범위를 조율하며, 변경사항을 디자인에 반영했습니다.
06

Process

기존 요구사항을 확인한 뒤 Tenant, Company, Data 관리 기능을 분류하고, List / Detail / Tab 기반 IA와 Data Ingestion 유저 플로우를 중심으로 화면 설계와 개발 검수를 진행했습니다.

01
Requirement Review
기존 요구사항과 기능 범위를 확인하고, Admin 관리 기능에서 필요한 화면 구조를 검토했습니다.
02
Management Feature Classification
Tenant, Company, Data 관리 기능을 구분하고 각 기능의 목적과 정보 구조를 정리했습니다.
03
List / Detail / Tab IA Design
Tenant Management를 중심으로 List, Detail, General, Store, Design 탭 구조를 설계했습니다.
04
SaaS Data Reference Research
SaaS 데이터 관리 솔루션을 리서치하며 Data Management 메뉴와 화면 흐름에 필요한 기준을 검토했습니다.
05
Data Ingestion User Flow
Data Ingestion 흐름을 중심으로 Admin 사용자가 데이터를 이해하고 관리할 수 있는 유저 플로우를 작성했습니다.
06
UX/UI Design Application
정리된 IA와 유저 플로우를 바탕으로 Tenant, Company, Data Management 화면 설계와 UI 디자인을 적용했습니다.
07
Development Review
개발팀 요구사항을 확인하고 기획 범위를 조율하며, 리뷰 과정에서 변경사항을 반영했습니다.

View / Edit QA
개발 완료된 Map View / Editor의 View/Edit 기능을 검수하고 구현 상태를 확인했습니다.

Key Work

SaaS Admin 환경에서 테넌트 정보를 효율적으로 관리할 수 있도록 Tenant Management 기능을 기획하고 화면을 설계했습니다. List와 Detail 구조를 중심으로 정보 탐색 흐름을 정리하고, 사업자 인증, Store Import, Login 커스터마이징 등 실제 운영에 필요한 관리 기능을 탭 구조로 구성했습니다.

  • —Tenant List 화면 기획 및 디자인
  • —Tenant Detail Page 세부 항목 기획
  • —General 탭 내 사업자 인증 관련 화면 설계
  • —Store 탭 내 Import Stores 기능 설계
  • —Design 탭 내 Login 화면 커스터마이징 설계
  • —개발 리뷰 대응 및 변경사항 반영
Data Management 기획 및 UX/UI 설계
복잡한 데이터 관리 구조의 IA 및 화면 설계

복잡한 데이터 관리 구조를 Admin 사용자가 이해하기 쉽도록 Data Management 메뉴와 화면 흐름을 설계했습니다. SaaS 데이터 관리 솔루션을 리서치하고, 개발팀과 데이터 구조 방향성을 조율하여 Data Ingestion 중심의 유저 플로우와 IA를 구체화했습니다.

  • —SaaS 데이터 관리 솔루션 레퍼런스 리서치
  • —STI / SSP 데이터 관리 구조 초안 요청 및 요구사항 확인
  • —Data Management 방향성 정리 회의 참여
  • —메뉴 기능 관련 개발 요구사항 확인
  • —변경된 IA 기반 메뉴 상세 설계
  • —Data Ingestion 유저 플로우 작성
  • —Data Management 화면 설계 및 디자인 적용
Company Management 개선 및 변경사항 반영
기획 누락 점검 및 일관된 UI 패턴 반영

Company Management 화면의 기획 누락 및 변경사항을 점검하고, UI 컴포넌트 구조를 정리했습니다. 기존 화면의 관리 기능 흐름을 유지하면서 변경된 요구사항이 일관된 UI 패턴으로 반영되도록 개선했습니다.

Before & After

Before

  • ✕Tenant, Company, Data 관리 기능의 IA가 명확히 정리되지 않음
  • ✕Tenant Detail 화면에서 정보를 어떤 탭으로 구분할지 기준이 필요함
  • ✕Data Management는 데이터 구조가 복잡해 사용자가 이해하기 어려울 수 있음
  • ✕Company Management 화면의 변경사항과 기획 누락 반영 필요
  • ✕개발 구현 결과와 디자인 산출물 간 검수 프로세스 필요

After

  • ✓Tenant, Company, Data 관리 기능의 IA와 화면 구조 구체화
  • ✓Tenant Detail을 General, Store, Design 등 탭 구조로 정리
  • ✓Data Ingestion 유저 플로우를 통해 데이터 관리 흐름 시각화
  • ✓Company Management 변경사항을 일관된 UI 패턴으로 반영
  • ✓View/Edit 기능 검수를 통해 디자인과 개발 구현 간 품질 차이 최소화

Outcome

Tenant, Company, Data Management 기능의 화면 구조와 관리 흐름을 구체화하여 SaaS Admin의 운영 기능 기반을 정리했습니다. 또한 개발팀과의 검토 및 기능 검수를 통해 디자인 산출물이 실제 구현 단계까지 연결될 수 있도록 기여했습니다.

Key Takeaway

이 프로젝트를 통해 SaaS Admin UX는 단순한 관리 화면 설계가 아니라, 조직·테넌트·데이터 구조를 사용자가 이해하고 운영할 수 있도록 정보 구조를 정리하는 일이라는 점을 경험했습니다. 특히 복잡한 데이터 관리 개념을 IA, 유저 플로우, 화면 단위로 구체화하고, 개발 검수까지 연결하며 Admin 제품 설계의 완성도를 높이는 과정을 수행했습니다.
