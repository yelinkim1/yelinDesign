SSP Design System
구축 및 운영 고도화

SaaS/Admin 제품군의 UI 일관성과 개발 효율을 높이기 위해 디자인 시스템의 컴포넌트 구조, 가이드라인 문서, 개발 연동 환경을 구축하고 고도화한 프로젝트입니다.

Role Product Designer
Design System Owner
Scope Component Structure
Guideline, Documentation
Dev Handoff
Tools Figma, Docusaurus
GitHub, Vercel
Duration 약 4주
7+ 디자인 시스템 개선 항목
6+ 문서화 / 가이드라인 작성 항목
10+ 구조 정리 대상 컴포넌트 그룹
1 개발 연동 환경 테스트
Background

기존 디자인 시스템은 컴포넌트 구조와 사용 기준이 일부 분산되어 있어, 디자이너와 개발자가 동일한 기준으로 컴포넌트를 사용하기 어려운 문제가 있었습니다.

SaaS/Admin 제품군이 확장되면서 컴포넌트의 역할, 사용 조건, 상태값, 문서화 기준을 일관되게 정리하고, 개발 문서와 연결 가능한 운영 기반을 마련할 필요가 있었습니다.

Problem

디자인 시스템이 존재했지만, 컴포넌트 구조와 문서화 기준이 일부 분산되어 있어 실제 제품 설계와 개발 협업 과정에서 동일한 기준으로 활용하기 어려운 상태였습니다.

01

컴포넌트 구조 분산

Foundation, Utility, Selection Control, Chart, Data Table 등 컴포넌트 그룹의 역할과 분류 기준을 재정리할 필요가 있었습니다.

02

사용 기준의 모호성

컴포넌트별 사용 조건, 상태값, Error 케이스, 예외 상황에 대한 기준이 명확하지 않아 화면마다 다르게 적용될 가능성이 있었습니다.

03

문서화 양식 불일치

컴포넌트 가이드라인의 Overview, Usage, Anatomy, Do / Don't 등 작성 구조를 통일할 필요가 있었습니다.

04

개발 연동 기반 부족

Figma 산출물이 실제 개발 문서와 연결되기 위해 Docusaurus, GitHub, Vercel 기반의 문서 사이트 환경 검토가 필요했습니다.

05

협업 프로세스 정리 필요

팀 공동 작업을 위해 GitHub Desktop 기반 Commit / Push 흐름과 문서 수정 프로세스를 정리할 필요가 있었습니다.

Goal

컴포넌트 구조를 재정리하고, 사용 기준을 문서화하여 디자인 시스템의 일관성, 재사용성, 개발 연계성을 높이는 것을 목표로 했습니다.

01

Improve Consistency

SaaS/Admin 제품군 전반에서 동일한 기준으로 UI 컴포넌트를 사용할 수 있도록 구조와 명칭을 정리합니다.

02

Clarify Usage Guidelines

컴포넌트의 역할, 상태값, 사용 조건, 예외 케이스를 문서화하여 협업 과정의 해석 차이를 줄입니다.

03

Build Documentation Base

Docusaurus 기반 문서 사이트를 통해 디자인 시스템이 웹 문서로 확장될 수 있는 기반을 마련합니다.

04

Connect Design and Development

Figma, GitHub, Vercel 환경을 연결하여 디자인 산출물과 개발 구현 간의 연동 가능성을 검토합니다.

My Role

Product Designer이자 Design System Owner로서 컴포넌트 구조 개선, Figma 기반 스타일 정리, 주요 컴포넌트 가이드라인 작성, Docusaurus 문서 사이트 구축 환경 테스트, GitHub / Vercel 연동 검토를 담당했습니다.

  • Component Structure Design — 기존에 분산되어 있던 컴포넌트 구조와 명칭 체계를 정리하고, 사용 맥락에 따라 컴포넌트를 재분류했습니다.
  • Figma System Organization — Figma 기반 컴포넌트, Variant, Text Style, Variable, Spacing 기준을 정리하고 일관된 구조로 재정의했습니다.
  • Guideline Documentation — 주요 컴포넌트의 사용 기준, 상태값, Error 케이스, 메시지 기준을 문서화했습니다.
  • Documentation Site Setup — Docusaurus 기반 디자인 시스템 문서 사이트를 생성하고 GitHub Repository 및 Vercel 배포 환경을 테스트했습니다.
  • Dev Handoff Review — 개발 구현 산출물과 디자인 산출물이 연결될 수 있도록 문서 구조와 개발 연동 가능성을 검토했습니다.
  • Collaboration Workflow — 팀 공동 작업을 위한 GitHub Desktop 기반 Commit / Push 프로세스와 문서 작업 흐름을 정리했습니다.
Process

기존 컴포넌트 구조를 분석한 뒤, 컴포넌트 그룹과 Foundation 기준을 재정리하고, 주요 컴포넌트 가이드라인과 문서 사이트 구축 환경을 순차적으로 정리했습니다.

01

Analyze Existing System

기존 디자인 시스템의 컴포넌트 구조, 명칭, 사용 기준을 점검하고 분산된 요소를 파악했습니다.

02

Reorganize Component Groups

Foundation, Utility, Selection Control, Chart, Data Table, Structured List 등 컴포넌트 그룹을 사용 맥락에 따라 재분류했습니다.

03

Define Foundation Standards

Text Style, Variable, Spacing, Padding, Gap 등 디자인 시스템 기반 요소의 기준을 정리했습니다.

04

Refine Component States

Radio, Checkbox, Switch 등 주요 컴포넌트의 상태값과 Error 케이스를 검토했습니다.

05

Write Usage Guidelines

Feedback, Loading, Pickers, Control Group 등 주요 컴포넌트의 사용 기준과 문서화 양식을 정리했습니다.

06

Build Documentation Site

Docusaurus 기반 디자인 시스템 문서 사이트를 생성하고, 문서 레이아웃과 작성 양식을 테스트했습니다.

07

Connect GitHub and Vercel

GitHub Repository와 Vercel 배포 환경을 연동하여 디자인 가이드라인이 웹 문서로 배포될 수 있는 기반을 검토했습니다.

08

Define Collaboration Workflow

GitHub Desktop 기반 Commit / Push 프로세스를 정리하고 공동 작업을 위한 데브 툴 사용 흐름을 정리했습니다.

Key Work

컴포넌트 구조 개선, 사용 기준 문서화, 문서 사이트 구축 환경 테스트를 통해 디자인 시스템이 실제 제품 설계와 개발 협업에 활용될 수 있는 운영 기반을 정리했습니다.

기존에 분산되어 있던 컴포넌트 구조와 명칭 체계를 정리하고, 사용 맥락에 따라 컴포넌트를 재분류했습니다. Foundation, Utility, Selection Control, Chart, Data Table 등 주요 컴포넌트 그룹의 역할을 명확히 하며 SaaS/Admin 제품군에 맞는 구조로 개선했습니다.

  • Text 구조 및 위계 정리
  • Text Style 기준 재정의
  • Foundation / Utility 컴포넌트 구조 정리
  • Selection Control 구조 재정리
  • Chart, Data Table, Structured List 컴포넌트 구조 개선
  • Spacing, Padding, Gap 관련 컴포넌트 사용 기준 정리
  • Radio, Checkbox, Switch의 상태값 및 Error 케이스 검토
  • Cursor, Tooltip, Grid Line, Bar Item, Node 등 차트 보조 요소 Utility화

컴포넌트 사용 기준을 디자이너와 개발자가 동일하게 이해할 수 있도록 가이드라인 문서를 작성했습니다. 컴포넌트의 역할, 상태값, 사용 조건, 예외 케이스를 문서화하여 협업 과정에서 발생할 수 있는 해석 차이를 줄였습니다.

  • 주요 컴포넌트 가이드라인 작성
  • Feedback, Loading, Pickers, Control Group 문서 작성
  • 컴포넌트별 Usage Guideline 정리
  • 상태값, Error 케이스, 메시지 사용 기준 검토
  • 디자인 시스템 문서의 레이아웃 및 작성 양식 통일
  • 등록된 Text Styles를 컴포넌트에 적용하며 스타일 일관성 검토

디자인 시스템이 단순한 Figma 파일에 머무르지 않고 실제 개발 문서로 확장될 수 있도록 Docusaurus 기반 문서 사이트 구축 환경을 테스트했습니다. GitHub와 Vercel을 연동하여 디자인 가이드라인이 웹 문서로 배포될 수 있는 기반을 마련했습니다.

  • Docusaurus 기반 디자인 시스템 문서 사이트 생성
  • GitHub Repository 생성 및 연결
  • Vercel 배포 환경 연동
  • 디자인 산출물과 개발 구현 간 연동 테스트
  • GitHub Desktop 기반 Commit / Push 프로세스 정리
  • 공동 작업 환경을 위한 데브 툴 프로세스 정리
Before & After
Before
  • 컴포넌트 구조와 사용 기준이 일부 분산되어 있음
  • 컴포넌트 명칭과 그룹 기준이 모호함
  • 상태값, Error 케이스, 예외 상황에 대한 기준이 부족함
  • 디자인 시스템 문서의 작성 양식이 통일되어 있지 않음
  • Figma 중심의 산출물이 개발 문서와 연결되기 어려움
After
  • 컴포넌트 그룹과 명칭 체계를 사용 맥락에 맞게 재정리
  • Text Style, Variable, Spacing 등 Foundation 기준 정리
  • 주요 컴포넌트의 Usage Guideline과 상태 기준 문서화
  • Docusaurus 기반 문서 사이트 구축 환경 테스트
  • GitHub / Vercel 연동을 통해 개발 문서와 연결 가능한 운영 기반 마련
  • 팀 공동 작업을 위한 Commit / Push 워크플로우 정리
Outcome

디자인 시스템의 컴포넌트 사용 기준을 명확히 하고, 향후 개발 문서와 연동 가능한 운영 기반을 마련했습니다. 컴포넌트 구조와 문서화 기준을 정리함으로써 디자이너와 개발자가 동일한 기준으로 시스템을 이해하고 활용할 수 있는 기반을 구축했습니다.

  • Component Structure Improved — 분산되어 있던 컴포넌트 구조와 명칭 체계를 정리하여 디자인 시스템의 일관성을 강화했습니다.
  • Usage Standards Clarified — 컴포넌트별 사용 기준, 상태값, Error 케이스를 문서화하여 사용 판단 비용을 줄였습니다.
  • Documentation Quality Improved — 디자인 시스템 문서의 작성 양식과 레이아웃을 통일하여 문서의 확장성과 가독성을 높였습니다.
  • Dev Integration Tested — Docusaurus, GitHub, Vercel 기반 문서 사이트 구축 환경을 테스트하여 개발 연동 가능성을 검토했습니다.
  • Collaboration Workflow Defined — GitHub Desktop 기반 Commit / Push 프로세스를 정리하여 팀 공동 작업 환경을 마련했습니다.
Impact

약 4주 동안 디자인 시스템 컴포넌트 구조, 가이드라인 문서, 개발 연동 환경을 정리하며 SaaS/Admin 제품군의 UI 일관성과 운영 기반을 강화했습니다.

Key Takeaway

이 프로젝트를 통해 디자인 시스템은 단순히 컴포넌트를 정리하는 작업이 아니라, 제품의 일관성과 개발 협업 효율을 높이는 운영 체계라는 점을 경험했습니다. Figma 컴포넌트 구조, 사용 기준 문서화, 개발 문서 연동 환경을 함께 고려하며 디자인 시스템이 실제 제품 개발 과정에서 지속적으로 활용될 수 있는 기반을 만드는 데 집중했습니다.
